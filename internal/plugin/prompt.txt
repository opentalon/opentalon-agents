Persistent agents: automations the user describes in chat, that you author ONCE in the Talon language and store here. They then run deterministically and autonomously — no model in the loop at run time.

Actions (all scoped to the current group):
- create(name, description, talon_source, [triggers]) — author a new agent. Set `description` to the USER'S ORIGINAL REQUEST in their own words (what they asked for), not a paraphrase of the Talon — this is how the human intent is preserved. Ask for a short name if they didn't give one. talon_source is validated before storing; if invalid you get compile diagnostics back — fix and retry.
- list / show(id) — inspect agents (show returns the full Talon source).
- run(id) — execute an agent's program now and return the result (good for testing).
- update(id, talon_source, [triggers]) — replace the program/triggers (re-validated).
- enable(id) / disable(id) / delete(id) — lifecycle.

For Talon syntax (workflows, on-blocks, `when`, facts, mcp steps, step("...").result), follow the Talon-language guidance from the talon plugin — it is authoritative. This section is about assembling an AGENT.

TWO KINDS OF AGENT
1. One-shot / batch — a plain `workflow` you trigger with run(id), or on a cron via a `schedule` trigger: [{"type":"schedule","cron":"0 9 * * 1-5"}] (standard 5-field cron, MINUTE granularity — no seconds field; never write a 6-field cron. For sub-minute cadence use a `poll` trigger with an interval like "30s" instead). Good for "do X every morning / across N items now".
2. Watcher — reacts to data changing over time: an `on`-block that fires a workflow when a value crosses a threshold, fed by a `poll` trigger. This is the shape for "monitor X and when Y happens, do Z".

AUTHORING A WATCHER
A watcher has two parts you provide together:

(a) talon_source — an `on change attr "<attr>"` block guarded by a `when`, firing a workflow that does the action. Use the fire-ONCE crossing guard so it acts on the transition, not every tick:

  on change attr "current_stock" {
    when prev_value >= 10 and new_value < 10
    workflow "Refill stock"
  }
  workflow "Refill stock" {
    step "ticket" {
      mcp "tickets" "create" {
        item step("trigger").result.entity
        qty  50
      }
    }
  }

The `poll` trigger does the FETCHING — do NOT call the polling MCP tool inside talon_source, and do NOT write a bare level check like `on current_stock < 10`. The engine polls the value and feeds it in as the fact your `on change attr` block watches; the only `mcp` step in talon_source is the ACTION taken when the block fires (here, creating the ticket). Author exactly a create(talon_source, triggers) — never a `pipeline` or other free-form structure, and never an `interval` trigger for a watcher (that is for schedules, not watched values).

(b) triggers — a JSON array with a `poll` entry telling the engine WHAT to fetch each interval and HOW to turn it into the fact the on-block watches:

  [{"type":"poll","config":{
    "server":"inventory","tool":"get-item","args":{"barcode":"ABC-123"},
    "interval":"5m",
    "value_path":"item.current_stock",
    "id_field":"item.barcode",
    "attribute":"current_stock"
  }}]

poll config fields:
- server / tool / args — the MCP tool to call and its static arguments.
- interval — how often, as a Go duration ("30s","5m","1h").
- value_path — dot-path to the watched value in the tool's response.
- id_field — dot-path to the entity's id (so multiple items stay distinct). Omit for a single watched thing.
- attribute — the fact attribute name. **It MUST match the `attr` in your `on change attr "..."` block** (here, "current_stock").
- items_path — OPTIONAL. To watch MANY entities from one poll, set this to the dot-path of the list in the response; value_path/id_field are then read from EACH item. The same on-block fires per item that crosses.

PUSH INSTEAD OF POLL (webhook)
Instead of polling, an external system can PUSH data. Use a `webhook` trigger — same `on`-block source, but the trigger carries just the mapping (no server/tool/interval):

  [{"type":"webhook","config":{"value_path":"stock","id_field":"barcode","attribute":"current_stock"}}]

The system then POSTs the data to `/v1/hooks/<agent-name>?user_id=<the user>` with header `Authorization: Bearer <webhook_secret>` (a shared secret the operator configures). The body is mapped to a fact exactly like a poll and evaluated on the next tick. Tell the user this URL if they want to drive the agent by webhook.

ESCALATION (fire → investigate → ask the user)
By default a watcher's reaction is a FIXED Talon action (the `mcp` step in its workflow), decided when you author it — no model at run time. Some tasks instead need judgement WHEN they fire: "stock is trending down — which items are at risk, and should I reorder or just alert you?". For those, opt the agent into escalation with the `escalate` argument on create/update:

  escalate: {"enabled":true}
  # or, with options:
  escalate: {"enabled":true,"prompt_template":"...","max_per_window":5,"window_seconds":3600}

When an escalation-enabled watcher fires, instead of only running its fixed action the system starts a full assistant turn in the USER'S session, seeded with what tripped (the agent's description = the user's original ask, the fired on-block, and the latest observed facts). That turn can investigate — including fanning out focused sub-agent checks — decide, and then message the user to ask how to proceed. Its reply is pushed to the user's channel, tagged as agent-originated.

Guidance:
- Escalation is OPT-IN and costs tokens (it runs a real LLM turn), so use it only when the firing genuinely needs judgement or a question back to the user. For a deterministic fixed reaction, author a normal workflow action and leave escalate off.
- Detection still stays deterministic: author the SAME `on change attr` + `poll`/`webhook` watcher. `escalate` only changes the REACTION. You still provide a talon_source; keep its workflow minimal (or a no-op notify) when the real work is the escalation turn.
- It is rate-limited (max_per_window / window_seconds; 0 = server default) and edge-triggered (fires on the crossing, not every tick), so a persisting condition won't spam the user.
- prompt_template optionally overrides the seed prompt; placeholders: {{agent_name}}, {{description}}, {{firings}}, {{facts}}.
- Escalation needs an interactive session to address the turn to — author it from the user's chat (enabling it from a non-interactive context with no known session is rejected).

RULES
- The `when` threshold must be a literal number (you can't compare against another fact). Bake the user's number in.
- The poll/webhook `attribute` MUST match the `attr` in your `on change attr "..."` block.
- `schedule` cron is 5-field, minute granularity. There is NO seconds field — a 6-field cron is invalid and will be rejected. For anything faster than a minute, use a `poll` trigger with a sub-minute `interval` ("30s").
- Keep one clear task per agent.
- Do NOT put the polling MCP call inside talon_source and do NOT emit a `pipeline`. A watcher is always create(talon_source = `on change attr` + workflow, triggers = a `poll` entry). The workflow's `mcp` step is the fire action only.
- Validate by create (invalid Talon is rejected with diagnostics) or by run(id).
